## Claude OAuth Token Fix (18:56-19:03 GMT)

### The Problem
- OAuth token in `auth-profiles.json` was `Symbol(clack:cancel)` — a JS error artifact from the CLI setup wizard
- The setup prompt was cancelled/errored and stored the cancel symbol instead of the real token
- OpenClaw tried it twice, got auth failures, put it on cooldown, then silently fell back to API key
- **Result:** David's Claude subscription was never actually being used despite the config looking correct

### The Fix
1. David provided the real OAuth token: `sk-ant-oat01-...` (starts with `oat01`)
2. Edited `/Users/claudia/.openclaw/agents/main/agent/auth-profiles.json` directly:
   - Set `profiles.anthropic:manual.token` to the real token
   - Cleared `usageStats.anthropic:manual` error count and cooldown
   - Set `lastGood.anthropic` to `anthropic:manual`
3. Backed up original to `auth-profiles.json.bak`

### Key File Location
- **Auth profiles:** `/Users/claudia/.openclaw/agents/main/agent/auth-profiles.json`
- **Config:** `/Users/claudia/.openclaw/openclaw.json` (has the profile definitions)
- Auth order in config: `auth.order.anthropic: ["anthropic:manual", "anthropic:default"]`

### What Didn't Work
- `gateway config.patch` → sends SIGUSR1 → soft reload does NOT reload auth-profiles.json
- `gateway restart` tool → also sends SIGUSR1 → same issue, doesn't reload tokens
- Running `openclaw gateway stop && openclaw gateway start` via exec → **CRASHED GATEWAY TWICE** — left David with no web UI, had to re-run wizard/onboard to recover
- Manually editing `auth-profiles.json` → caused crash on gateway restart
- First token (`sk-ant-oat01-Uxa9N...`) was invalid/expired — testing with curl confirmed 401 auth error
- Browser session key (`sk-ant-sid02-...`) doesn't work as API auth

### What DOES Work (CORRECT PROCEDURE — CONFIRMED 20:01 GMT)
1. `claude setup-token` — generates a fresh OAuth token (valid 1 year)
2. `openclaw onboard --auth-choice setup-token` — **THE ONLY METHOD THAT WORKS**
   - Choose "existing values" (NOT reset)
   - Paste the `sk-ant-oat01-...` token when prompted
   - Leave gateway token as-is
   - Install gateway service (yes)
   - Gateway restarts automatically
3. Verify with `session_status` — look for `🔑 token sk-ant…JwAA (anthropic:manual)`

### What Does NOT Work (Tried and Failed)
- `openclaw models auth paste-token --provider anthropic` — stores token but gateway ignores it
- `openclaw models auth setup-token --provider anthropic` — same problem
- Manually editing auth-profiles.json — caused gateway crashes
- Changing mode between "token" and "oauth" in config — neither fixes it
- Running gateway restart from exec tool — crashed gateway twice

### Lessons Learned
- **SIGUSR1 soft restarts do NOT reload auth-profiles.json** — only openclaw.json changes
- **Use `openclaw gateway restart`** (single command) — NOT `stop && start` (can leave gateway dead)
- **NEVER run gateway restart from exec tool** — it kills the session and can crash the gateway
- Always verify auth after changes with `session_status` — check the 🔑 line
- `Symbol(clack:cancel)` bug: if `openclaw configure` is interrupted, check auth-profiles.json manually
- OAuth tokens are setup-tokens from `claude setup-token` (Claude Code CLI), NOT browser session keys
- Token format: `sk-ant-oat01-...` (OAuth) vs `sk-ant-api03-...` (API key) vs `sk-ant-sid02-...` (session, unusable)

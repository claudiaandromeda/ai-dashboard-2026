## LOGGING ARCHITECTURE REDESIGN (11:35 GMT) — CRITICAL

### Problems with Current Setup (Haiku Subagent)
1. **mode=run** (one-shot) — if agent crashes, logging dies silently
2. **No restart logic** — failure = no alert, no recovery
3. **No fallback** — if Haiku API fails, conversation unlogged
4. **No health check** — David won't know logging is broken until next session
5. **Cost** — Haiku ~$0.02/day (small but unnecessary)

### Bulletproof Architecture (David Approved)

**5-Layer Redundancy:**
1. **Primary:** Local Ollama agent (mistral:7b) reading session history every 5 min
   - Zero cost, instant, zero latency
   - Writes to `/memory/conversation-logs/2026-03-03_emotivx-rebuild.md`
   
2. **Fallback 1:** Python file-append script (direct write, no model needed)
   - If Ollama unavailable, Python appends messages directly
   - No intelligence needed, just file I/O
   
3. **Fallback 2:** Git auto-commit hourly
   - All logs committed to git
   - Recoverable even if disk fails
   
4. **Health Check:** Cron job every 10 minutes
   - Check timestamp of last log update
   - If > 10 min stale → Telegram alert to David
   - Alert includes: last 3 messages + "logging down" status
   
5. **Dashboard:** Status page showing:
   - ✅ Healthy (updated < 5 min ago)
   - ⚠️ Stale (5-10 min old)
   - ❌ Failed (> 10 min old)
   - Message count today
   - Backup status

### Why This Works
- **No silent failures** — David is alerted within 10 minutes if anything breaks
- **No cost creep** — Local Ollama is free
- **Extreme redundancy** — needs 3+ things to fail simultaneously for data loss
- **Git recovery** — even if local disk fails, history in git
- **Dashboard visibility** — David knows system health at a glance

**Status:** APPROVED — Ready to implement
